# وثيقة المتطلبات التفصيلية للمنتج الأولي القابل للتطبيق (MVP)

## 1. مقدمة

### 1.1. رؤية المشروع (لـ MVP)
تتمثل رؤية المنتج الأولي القابل للتطبيق (MVP) في إطلاق منصة B2B أساسية وفعالة تربط بين المهنيين البيطريين (الأطباء، العيادات، المزارع) والموردين في مصر، مع التركيز على توفير الوظائف الأساسية للشراء والبيع وإدارة العمليات الأولية.

### 1.2. أهداف الـ MVP
- **التحقق من السوق:** التأكد من وجود طلب حقيقي على منصة مركزية للمستلزمات البيطرية في السوق المصري.
- **اكتساب المستخدمين الأوائل:** جذب مجموعة أولية من المشترين والبائعين لاختبار المنصة وتقديم ملاحظاتهم.
- **بناء أساس تقني متين:** إنشاء بنية تحتية قابلة للتطوير والتوسع في المستقبل لإضافة الميزات المتقدمة.
- **تقليل وقت الوصول للسوق (Time-to-Market):** إطلاق نسخة أولية بسرعة لبدء العمليات وجمع البيانات.

## 2. أدوار المستخدمين (User Roles)

1.  **المشتري (Buyer):** الطبيب البيطري، مدير العيادة، مدير المزرعة.
2.  **البائع (Seller):** المورد، الشركة المصنعة، الموزع المعتمد.
3.  **المسؤول (Admin):** مدير المنصة المسؤول عن الإشراف والتحكم.

## 3. المتطلبات الوظيفية (Functional Requirements)

### 3.1. متطلبات عامة
- **الصفحة الرئيسية:** عرض بسيط للمنتجات المميزة أو الفئات الرئيسية.
- **صفحات محتوى أساسية:** "من نحن"، "اتصل بنا"، "الشروط والأحكام".

### 3.2. متطلبات المشتري (Buyer)

| الميزة | الوصف التفصيلي |
| :--- | :--- |
| **التسجيل وإنشاء الحساب** | - التسجيل باستخدام البريد الإلكتروني وكلمة المرور.<br>- إدخال معلومات أساسية (الاسم، اسم العيادة/الشركة، رقم الهاتف، العنوان).<br>- يجب أن تتم الموافقة على الحساب من قبل المسؤول قبل تفعيله. |
| **تسجيل الدخول** | - تسجيل الدخول باستخدام البريد الإلكتروني وكلمة المرور.<br>- خيار "تذكرني" و "نسيت كلمة المرور". |
| **إدارة الملف الشخصي** | - عرض وتعديل المعلومات الشخصية ومعلومات العيادة.<br>- إدارة عناوين الشحن. |
| **تصفح وبحث المنتجات** | - البحث عن المنتجات بالاسم أو الكلمة المفتاحية.<br>- تصفية المنتجات حسب الفئة الرئيسية.<br>- عرض قائمة المنتجات مع صورة مصغرة، اسم، وسعر. |
| **صفحة تفاصيل المنتج** | - عرض صورة المنتج، الاسم، الوصف، السعر، اسم البائع، والمواصفات الأساسية. |
| **عربة التسوق** | - إضافة المنتجات إلى العربة.<br>- تعديل كمية المنتجات أو حذفها.<br>- عرض ملخص الطلب (الإجمالي الفرعي). |
| **إتمام الطلب (Checkout)** | - عملية من خطوة واحدة أو خطوتين (عنوان الشحن، اختيار طريقة الدفع، تأكيد الطلب).<br>- اختيار طريقة الدفع (دفع عند الاستلام، تحويل بنكي، بوابة دفع واحدة مثل Paymob). |
| **سجل الطلبات** | - عرض قائمة بالطلبات السابقة والحالية.<br>- عرض تفاصيل كل طلب (المنتجات، السعر، حالة الطلب). |

### 3.3. متطلبات البائع (Seller)

| الميزة | الوصف التفصيلي |
| :--- | :--- |
| **التسجيل وإنشاء الحساب** | - عملية تسجيل مشابهة للمشتري ولكن مع حقول إضافية (اسم الشركة، السجل التجاري).<br>- يجب أن تتم الموافقة على حساب البائع من قبل المسؤول. |
| **إدارة الملف الشخصي** | - عرض وتعديل معلومات الشركة ومعلومات الاتصال. |
| **إدارة المنتجات** | - لوحة تحكم لإضافة، تعديل، وحذف المنتجات.<br>- إدخال تفاصيل المنتج (الاسم، الوصف، السعر، الكمية المتاحة، الصور، الفئة). |
| **إدارة الطلبات** | - عرض قائمة بالطلبات الواردة لمنتجاته.<br>- تحديث حالة الطلب (قيد المراجعة، تم الشحن، مكتمل). |

### 3.4. متطلبات المسؤول (Admin Panel)

| الميزة | الوصف التفصيلي |
| :--- | :--- |
| **لوحة التحكم الرئيسية** | - عرض إحصائيات أساسية (عدد المستخدمين، عدد الطلبات، إجمالي المبيعات). |
| **إدارة المستخدمين** | - عرض قائمة بجميع المستخدمين (مشترين وبائعين).<br>- الموافقة على الحسابات الجديدة أو رفضها.<br>- حظر أو تفعيل حسابات المستخدمين. |
| **إدارة المنتجات** | - عرض جميع المنتجات على المنصة.<br>- إدارة الفئات الرئيسية للمنتجات (إضافة، تعديل). |
| **إدارة الطلبات** | - عرض جميع الطلبات التي تتم على المنصة.<br>- البحث عن طلبات محددة وتتبع حالتها. |

## 4. المتطلبات غير الوظيفية (Non-Functional Requirements)

| النوع | الوصف |
| :--- | :--- |
| **الأداء (Performance)** | - يجب أن يتم تحميل الصفحات الرئيسية في أقل من 3 ثوانٍ. |
| **الأمان (Security)** | - حماية كلمات المرور (Hashing).<br>- حماية أساسية ضد هجمات الويب الشائعة (XSS, SQL Injection).<br>- استخدام HTTPS لتشفير جميع الاتصالات. |
| **قابلية الاستخدام (Usability)** | - واجهة مستخدم بسيطة ونظيفة وسهلة التنقل.<br>- تصميم متجاوب (Responsive) يعمل بشكل جيد على أجهزة الكمبيوتر المكتبية والهواتف المحمولة. |
| **الموثوقية (Reliability)** | - يجب أن تكون المنصة متاحة بنسبة 99% من الوقت. |
| **قابلية التوسع (Scalability)** | - يجب أن يتم تصميم البنية التحتية للسماح بإضافة ميزات جديدة وتوسيع قاعدة المستخدمين في المستقبل دون إعادة بناء كاملة. |
